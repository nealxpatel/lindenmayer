# 9.1 Core foundation library: event model, kinds, relay, verification

## Goal

Ship all six contract deliverables in `src/lindenmayer/core/` with tests and
the nine `docs/kinds/` files, per the countersigned contract (95D09C90).

## Architecture (frozen interface for children)

```
src/lindenmayer/core/
  __init__.py        # public API re-exports
  event.py           # NIP-01 Event: canonical id, serialize; frozen model
  keys.py            # secp256k1 keypair, BIP340 Schnorr sign/verify
  kinds/             # typed models, one module per kind + registry (child: kinds)
  relay.py           # NIP-01+29+42 client, capability-gated privacy (child: relay)
  verify.py          # §6.5 verification: attestation chains, approval counts,
                     # revocation filtering, extraction labeling (child: verify)
  config.py          # relay URL, key material, capability attestations (mine)
  README.md          # library doc incl. acceptable-degradation posture (mine)
tests/               # mirrors modules; NIP-01 vectors in tests/vectors/
docs/kinds/          # nine files (child: kinds)
```

Interface freeze rule: `event.py`, `keys.py`, `config.py` are MINE and land in
the skeleton commit children fork from. Children build against `Event` (raw
events + tag access), never against each other's unmerged code. `verify.py`
consumes raw `Event`s and tag semantics from event-kinds.md — not the kinds
child's pydantic classes; I wire typed integration at merge.

## Runtime dependencies (justification, per constraint)

- **coincurve** — BIP340 Schnorr over libsecp256k1 (bitcoin-core's C library,
  the BIP340 reference implementation). The trust root should not ride a
  pure-Python curve implementation; coincurve is the maintained binding.
- **pydantic v2** — typed kind models with validation; the accept/reject
  test surface the contract demands is pydantic's core competence.
- **websockets** — mature asyncio WebSocket client for the relay client.
Dev-only: pytest, pytest-asyncio.

## Delegation decision (out loud)

Triggers fire: three separable, parallelizable subtasks with real depth, and
the kinds subtask wants a clean context (23K event-kinds.md + three upstream
NIP drafts). Spawn 3 children after the skeleton commit:

1. **kinds** (~$8 cap): `kinds/` package + `docs/kinds/` nine files +
   accept/reject tests per kind. Reads event-kinds.md deeply; pulls
   NIP-OA/AM/AO from block/buzz (I will cache copies if I fetch first).
2. **relay** (~$7 cap): `relay.py` + mock-relay tests. NIP-01+29+42 only;
   private-group gating refused absent a capability attestation (fail loud).
3. **verify** (~$6 cap): `verify.py` + tests for valid/expired/revoked
   attestation chains, approval-count edges, extraction-time labeling.

All leaf-capped (`--max-depth=0 --max-children=0`), disjoint file ownership in
NODE.md text (shared scope dirs are directory-granular). Budget both sides:
~$21 children + spawn ceremony + ≥1 integration iteration for me fits $32.6
remaining; my own burn ≈$1/step so far.

Mine, not delegated: event model + keys (trust root, part of the frozen
skeleton), config.py, core README degradation posture, collision check +
architect escalation, integration/merge, final test.sh green.
