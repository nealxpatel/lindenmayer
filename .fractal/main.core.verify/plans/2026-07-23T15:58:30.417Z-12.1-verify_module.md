# 12.1 verify.py: attestation, approvals, revocation, labeling

Leaf node, no children (max_children=0) — solo build, no delegation.

## Design decisions (from spec reading)

**validate_attestation(event) -> AttestationResult**
- `AttestationOutcome` enum: `VALID`, `ABSENT` (no auth tag), `INVALID`
  (tag present but fails — duplicate tag, malformed length, bad hex,
  self-attestation, bad grammar, sig fails, or conditions unsatisfied by
  this event).
- Order: `event.verify()` first (else `INVALID`/reason `invalid_event`) →
  count `auth`-named tags in `event.tags` directly (0 → `ABSENT`; >=2 →
  `INVALID`/`duplicate_auth_tag`) → exactly 4 elements → hex validation
  (owner 64/lower, sig 128/lower) → self-attestation → STRICT conditions
  grammar (own regex, not `str.isdigit()` — must reject unicode digits and
  leading zeros) → preimage/SHA256/BIP-340 verify via
  `keys.schnorr_verify` → evaluate each clause against the event (AND
  semantics) → `VALID`.
- Reason is a short machine-readable snake_case string reused verbatim by
  `attestation_state`.
- Conditions grammar: split on `&`; empty string valid (no clauses); any
  empty split segment (leading/trailing/double `&`) rejects; each clause
  fullmatch `^(kind=|created_at<|created_at>)(0|[1-9][0-9]*)$`; range check
  (kind 0-65535, created_at 0-4294967295).

**count_approvals(request, events) -> ApprovalCounts (NamedTuple: approve_count,
reject_count, per_approver dict[pubkey, "approve"|"reject"])**
- Kind-42041 verdict schema is NOT yet specified anywhere reachable (no
  docs/kinds/ on this branch yet — sibling's in-flight work). Decision:
  use `content` field, exactly `"approve"` or `"reject"`; anything else is
  not a valid verdict (excluded, not counted). Will radio main.core
  (priority 5, design note) in EXECUTE so the kinds/relay siblings can
  reconcile or override at merge — not blocking, documented assumption.
- Valid verdict candidate: kind==42041, `event.verify()`, has an `e` tag
  whose value equals request.id, content in {approve, reject}.
- Per-approver: latest wins by (created_at desc, id asc as tiebreak).
- `is_approved(request, events, threshold, required_approvers=None)`:
  filter per_approver to required set if given, count approves, compare
  to threshold.

**filter_attested / attestation_state**
- `attestation_state` computes: `INVALID_EVENT` if `event.verify()` fails;
  else run `validate_attestation`; ABSENT outcome -> check `event.pubkey
  in revoked_agents` (agent ban applies even without an auth tag) ->
  `REVOKED` else `UNATTESTED`; VALID outcome -> check owner in
  revoked_owners or event.pubkey in revoked_agents -> `REVOKED` else
  `ATTESTED`; INVALID outcome -> `INVALID_ATTESTATION` (carry the reason
  string).
  Document this "agent revocation applies regardless of attestation"
  choice explicitly in the docstring — it's a deliberate extension beyond
  the literal enum description, made so the reader-side trust bound
  actually bounds the agent key, not just the attestation claim.
- `filter_attested` keeps events whose `attestation_state` is `ATTESTED`
  or `UNATTESTED`; drops `INVALID_EVENT`, `INVALID_ATTESTATION`, `REVOKED`.
  Docstring caveat: latency — a plain relay still serves stale/revoked
  events; this is the reader-side bound, not relay enforcement.
