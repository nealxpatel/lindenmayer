# 30.1 Build evergreen requirements inventory

## Scope

Decompose `tree/root/CONTEXT.md` into discrete requirements that the evergreen surface must generate/maintain, classify by data source, survey existing surfaces, and map to four sec-5 capabilities. Deliverable: `docs/research/evergreen/inventory.md` with requirements table, source classification, existing-surface survey, and per-capability gap list.

## Approach

1. **Decompose CONTEXT.md sections** into discrete requirements, each asking "what must an evergreen surface generate, maintain, or show":
   - Why this file exists (design principle)
   - Standing context (mission, phase, non-negotiables)
   - How the root steers (radio/live steering/approvals/budgets)
   - Model policy (role → model tier mapping)
   - Merge hygiene (safeguards and checks)
   - Consultation covenant (governance contract)
   - Pointers (key references)

2. **Classify by data source**:
   - *relay-derivable*: from Lindenmayer relay events (NIP-01/29/42, custom kinds 42010/42020/42030/42040/42041/42050/38110/38150)
   - *fractal-derivable*: from Fractal SQLite or CLI surfaces
   - *human-authored*: standing context no query produces

3. **Survey existing surfaces**:
   - bridge CLI: `src/lindenmayer/bridge/cli.py`
   - registry CLI/reader: `src/lindenmayer/registry/cli.py`, `reader.py`
   - SQLite adapter: `src/lindenmayer/bridge/adapters/sqlite.py`
   - Test relay connectivity (ws://localhost:7100) read-only

4. **Map to sec-5 capabilities**:
   - Commission trees on the fly
   - Steer subgraphs
   - Review approvals
   - Query own history
   
   For each: which requirements/data exist today and what gaps remain.

5. **Synthesize findings** into the inventory table and per-capability gap analysis.
