## Decision 2 — Compaction-to-task mapping (requester: root, directive 7803C28A; §8 → §5)

The child recommended Candidate A (a distinct compaction event, task
coordinates in tags, `e`-tag `summary-of` marker, metrics-only content).
**Adopt the shape; reject one of its four next-steps; add a detection ruling
it did not reach.**

- **Adopt** Candidate A's shape. Signed, discoverable, privacy-tight, and
  harvestable inside the one adapter (bridge condition 3, verdict 8266A685).
- **Reject next-step 3** — "coordinate with Claude Code's harness to emit a
  transcript record type `compaction`". A design surface may not depend on an
  unshipped change to an external platform we do not control. Direct
  precedent: decision row 313 endorsed the buzz kind-allowlist PR as a
  parallel track while ruling that *no design surface may depend on its
  acceptance*; §6.3 is the same instinct applied to Fractal. The
  recommendation as written is contingent on a marker that does not exist, so
  it is unbuildable.
- **Detection ruling (mine, not the child's).** Compaction is already
  observable in the data the harvester reads today: within one session the
  per-request context size (`input_tokens` + `cache_read_input_tokens`) grows
  monotonically and *resets sharply* at a compaction. So compaction is
  detectable as a **usage-series discontinuity** — no new transcript record
  type, no upstream dependency, no conversational parsing (honoring bridge
  condition 3's "append-only per-request usage fields only"), and it lands in
  exactly one module (constraint b).
- **Provenance grade on the event.** Because detection is an inference rather
  than an attested fact, the event carries a `detection` tag —
  `usage-discontinuity` today, `harness-marker` if one ever ships. A reader
  can then tell a derived claim from an attested one. This is §6.5's
  verify-don't-assume posture applied to our own telemetry.
- **Kind allocation: 42060.** State the convention that makes it mechanical:
  the 420xx regular block is decade-per-family (42010 lifecycle, 42020
  accounting, 42030 digest, 42040/42041 approval, 42050 template), with
  intra-family increments in the units digit. Session compaction is its own
  family, so it takes the next decade. *Condition:* `core` re-runs the
  block/buzz collision check before writing `docs/kinds/42060-*.md` (nearest
  Buzz neighbour was 42000 @ 06e3d82; 42060 is expected clear, but the check
  is core's to run and mine to require).
- **§6.1 compliance:** content carries metrics and a **summary hash, never
  summary text**; tags carry task coordinates only. The pointer resolves to
  the node's *own* step event, so a child's compaction event stays in its
  subgraph — only digests roll up.

This **closes** the §8 open question: it moves from §8 into §5 as current

## Decision 3 — What evergreen IS at v1 (requester: root, directive 7803C28A; §5)

**Ruling: evergreen v1 is the read plane; Fractal's own CLI remains the write
plane.** Concretely, v1 is (a) a query surface over the signed log and (b) a

## Post-Mortem

### Accomplishments

All four decisions landed in `docs/DESIGN.md` with decision-log rows naming
requester and verdict: model-policy retiering (approve), compaction (§8
closed, now §5.2), evergreen v1 (read plane, §5.1), Buzz v1 (§1). The research
aggregate was rewritten from the in-progress draft into a real aggregate. The
kind-allocation convention and cross-post rules were promoted to the shared
wiki as `event_kind_conventions.md`, pointing at DESIGN.md as canonical rather
than duplicating it. Saved radio queue drained to empty. No children spawned,
per the delegation analysis above, and that judgment held.

### Deviation: the compaction ruling was wrong, and REVIEW caught it

The plan's Decision 2 rested on the child's finding that no compaction marker
exists in transcripts. Verifying that premise at REVIEW — one script over
`transcripts/` — showed it false. The harness already writes a `system` record
with `compactMetadata` (`trigger`, `preTokens`, `postTokens`, `durationMs`,
`cumulativeDroppedTokens`, preserved-segment ids), plus a `user` record flagged
`isCompactSummary`; two markers across 36 transcripts, in exactly the two
sessions that compacted.

So the ruling's entire reasoning — reject the upstream dependency, substitute
an inference — was sound argument on a false premise. The upstream dependency
never needed incurring.

The substituted inference was independently wrong in detail. The two-term
series I published (`input_tokens` + `cache_read_input_tokens`) tracks cache
turnover rather than context: cache expiry collapses `cache_read` while
`cache_creation` absorbs the prompt. It fired 4–6 false positives per session
across the four largest sessions. The three-term form (adding
`cache_creation_input_tokens`) fired twice, in exactly the two marker-bearing
sessions — 1:1.

Both errors had already shipped into DESIGN.md, the aggregate, the wiki page,
and a priority-6 radio message to the root for bridge's implementer. All four
surfaces were corrected, a superseded-finding note was added at the head of
the child's study so a direct reader is not misled, and a correction went out
(31BA28A8).

One thing the correction improved rather than merely fixed: because the
harness writes summary *text* into the transcript and `compactMetadata`
carries preserved-message UUID lists, the "summary hash, never summary text"
rule stopped being a theoretical §6.1 guard and became a live one. Bridge
condition 3 is now explicitly extended — append-only structured metadata is
readable, conversational structure and the summary body are not.

### Note for the next run

The generalizable rule, now in memory: **a child's negative existence claim
("no X exists") is a premise, not a finding, and gets verified before any
ruling rests on it.** The check cost one script; the error had already
propagated to four surfaces and another node's implementer. Positive findings
a child reports are cheap to trust because they cite evidence; negative ones
assert the absence of evidence, and absence is exactly what a narrow search
produces by accident.
