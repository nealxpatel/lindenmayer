## Part 1 — Design review (solo; governance verdicts are mine to give)

Proposal: make standard Nostr the PRIMARY integration target — core telemetry
consumable by any compliant Nostr client/relay — with Buzz-specific kinds and
conveniences as a layer on top. Revises §1, which currently targets Buzz
events directly.

Preliminary assessment (to be finalized in EXECUTE): approve. The proposal
strengthens principle §6.2 (the signed event log is the history — an event
log readable only by one v0.4.x product is a weaker source of truth) and
adds a churn hedge at zero capability cost (Buzz remains the preferred human
surface). The hard part the root flagged: classifying Buzz NIP *drafts*.

Working classification rule: **core may depend on a Buzz NIP draft only in
its schema-on-the-wire aspect — signed events any compliant relay can store
and any client can read from the documented kind — never in its relay-side
enforcement/behavior aspect.** Applied:

- NIP-OA (owner attestation): attestation *events* are core (they are just
  signed events; the draft is public); relay-enforced instant access
  revocation is Buzz-layer. Without Buzz, attestation degrades to verifiable-
  but-advisory. §2 wording gets a touch-up to match.
- NIP-AM/AO (metrics): schema alignment allowed, but core kinds must be
  self-documented so they survive draft churn (feeds the kinds research).
- Branch-as-room, buzz-protect: relay behavior → Buzz-layer only.

Actions: rewrite §1 as two-layer (Nostr core / Buzz layer), touch §2, add
decision-log row (requested by root, citing 5CD20B25), reply verdict on the
thread, then **mid-iteration `fractal commit`** so children fork a branch
that already contains the adopted layering.
