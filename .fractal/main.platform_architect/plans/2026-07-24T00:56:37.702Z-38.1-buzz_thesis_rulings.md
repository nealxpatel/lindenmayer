## The rulings

### R1 — NIP-01 tag filters (`6D3D3C19`). Diagnosis APPROVED; general rule to §6.5.

Confirm the diagnosis and the client-side fix; direct `registry` to apply the
same one-line shape. Elevate to a **corollary of principle 5** rather than two
call-site patches: a relay filter is a bandwidth optimization, never a
correctness boundary; only NIP-01 single-letter `#<x>` keys may be sent at
all, because a non-conformant key is *silently ignored* rather than rejected —
it returns a superset that looks filtered. Add the standing prohibition on
loosening shared test doubles (a double more permissive than the strictest
conformant real implementation masks this class tree-wide). Confirm the §6.1
reading: a branch-labeled surface carrying twelve other subgraphs is an
aggregates-up violation, hence blocking, not cosmetic.

### R2 — (a) §0 thesis wording. Change it; it is a **correction of a lag, not a revision**.

The Nostr-first layering (`5CD20B25`) already made Nostr the primary target
and Buzz a layer. §0 has been lagging its own architecture ever since. Framing
it as a correction matters: it is why nothing built is at risk. Proposed body
in EXECUTE; root applies the parallel edit to repo `CLAUDE.md`.

### R3 — (b) does demoting Buzz weaken §6.1's integrated-graph framing?

Make the strongest case AGAINST first, as asked: "integrated" is load-bearing;
agent work in the org's real inboxes makes agents colleagues, while a bespoke
dashboard makes them a fleet you monitor. Sharpest form — one the directive
did not make — is that §1 names **channel membership as the only access
gate**, so a custom UI appears to require rebuilding the very gate §6.1 calls
structurally enforced.

Then resolve it: the gate is **NIP-29**, a *relay* feature, not a Buzz-client
feature. A custom UI reading the same relay inherits the same gate. Every
mechanism §6.1 relies on — membership gating, aggregates-up by kind schema,
one-hop radio topology — lives in the protocol, the relay, or Fractal. **None
lives in the Buzz client.** The integrated-graph claim is carried by shared
identity and a shared log, not a shared app; "any Nostr client, including
Block's" is a *stronger* interop claim than "our app." Honest residual: ambient
presence is the one thing a UI cannot replicate, and that residual is exactly
what ranks NIP-AO in R5 rather than killing it.

### R4 — (d) approvals as a write-plane exception. **Reject for v1**, on different grounds than anticipated.

Not because signing is wrapping. Because making the signature *consequential*
requires a relay→Fractal control path (a second, automated control path —
precisely §5.1/§6.3), while leaving it inconsequential creates a second,
disagreeing record of whether work was approved (§6.2). Both horns are worse
than the status quo. Name the condition that would flip it: Fractal itself
gaining a native signed-log approval input, at which point the control path is
Fractal's own contract and not our wrapper. A deferral with a trigger, not a
permanent no. Consequence: **the UI is read-only at v1.**

### R5 — (c)+(f) collapsed. *Provisional pending the child.*

One generator, several rendering targets: signed log → query surface →
{markdown context surface, UI, optionally NIP-AO frames}. Adding a target is
the generator doing its job, not a new component ⇒ **evergreen v2, not a new
ply node** (a second node would need commit rights over the same query
surface — worse governance). NIP-AO is **Buzz-layer** under §1's
draft-dependency rule, and is independently disqualified from the core log by
being *ephemeral by spec* — §6.2 settles it before the layering argument runs.
Ranking: **UI first, NIP-AO second and optional**, owned by `bridge` (it is a
derived outward-publishing path, which is bridge's existing half) rather than
evergreen.

Marked provisional because it hinges on the claim the child is verifying. If
the child refutes "Buzz cannot render the tree," the ranking flips and the UI
charter narrows.

### Deviations from plan

**One unplanned finding, and it was the most valuable output of the run.**
REVIEW's instruction to re-verify printed claims caught a *false claim already
in DESIGN.md* — §8 asserted no relay is reachable from inside a node run, and
had used that to justify a tree-wide rule barring completion gates on live
relay data. Measured: `nc -z localhost 7100` succeeds. The relay had been
answering the whole time the claim stood.

Chasing it surfaced a second, independent defect in `registry`:
`registry/cli.py:35` defaults `--relay` to `ws://localhost:8080`, a closed
port, and it is the *only* relay default anywhere in the tree.

§8 was rewritten to gate on **reproducibility, not reachability** — the rule
survives, its justification changed — and the stale claim was corrected in the
shared wiki too, since other nodes were reading it there.

The plan also under-specified verification: it treated re-verifying
`6D3D3C19` as the whole job, when the broader duty was re-deriving *every*
printed claim. The first tag-filter sweep used one regex; REVIEW re-ran it
across both quote styles and f-string/concatenation construction before the
one-line blast-radius claim was allowed to stand in three documents. It held.
