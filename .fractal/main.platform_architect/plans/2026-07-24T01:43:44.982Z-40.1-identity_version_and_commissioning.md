Two priority-6 root directives, ruled as one pass: `35718844` (identity ×
version policy; evergreen v2 commissioning) and `2A25343E` (§1 revocation
defect; evergreen absorbs intake; @mention reopened). Receipt sent as
`24420800`.

The root asked for judgment over research. Budget is not the constraint
($24 remaining); the work is judgment-shaped, not survey-shaped.

## The rule the whole pass turns on

The root named the crux itself: *two records that can disagree*. Sharpened, it
generalizes past this directive and settles all four open write-path questions
at once:

> **A new signed record is safe exactly when Fractal holds no competing
> assertion of the same proposition.** Where Fractal already asserts the
> proposition, a second record can diverge from the one that actually governs
> execution, and something must reconcile them — a second control path (§6.3).
> Where Fractal asserts nothing, the signed record is the *only* record, and
> divergence is impossible because there is nothing to diverge from.

Applied:

| Path | Fractal's competing assertion | Verdict |
|---|---|---|
| Approvals (42041 → gate) | gate state — approved/not | **barred** (already ruled; rule confirms it) |
| radio-send / signals / kill wrappers | the message, the signal, the kill | **barred** (already ruled) |
| Commission authorization | *none* — Fractal has no concept of authorization | **permitted** |
| Inbound @mention → radio | *none* — the mention's origin is outside Fractal | **permitted** |

The sharp line against approvals survives and gets sharper:
**authorization-without-execution is a coherent state; approval-without-release
is not.** A commission that was never spawned is an unexecuted authorization,
which reads true. A verdict that never released the gate is a record claiming
work was approved while the work sits blocked — a record that lies.

## Rulings to write

### A. Identity × version (`35718844` item 1) — approve (a)–(d), (e) with a condition

- (a) role-scoped identity, (b) explicit recorded adoption, (c)
  revision-keeps / fork-mints with the inherit `e`-tag as design ancestry not
  reputation transfer, (d) version-scoped trust over role-scoped identity —
  all approved as proposed.
- (e) kind-10100 profile advertising current version — approve **as a derived
  pointer, never the record**. 10100 is replaceable, which is correct for
  "current" but fatal for history; and a version *restated* inline in a
  profile is the same stale-restatement failure §3 already records against the
  tier table. Condition: the profile carries resolvable coordinates (`a`-tag
  to the 38150 pointer plus the adopted 42050 event id), not a transcribed
  version string. Buzz-layer surface; the adoption event is the core fact.
- Key-role separation **confirmed**: templates signed by their human author,
  nodes signed by their own keypairs and attested via NIP-OA. Tonight's three
  42050s signed with the owner key are correct — a template version *is*
  human-authored.

### B. §1 revocation defect (`2A25343E` item 1) — accepted as a defect

- (a)–(d) approved. Attest subgraphs to a durable org/seat key; a human's
  evergreen node holds a revocable grant; succession moves grants, not
  fractals; the evergreen node itself is non-transferable.
- **The argument for (a) the root did not make, and the strongest one:**
  NIP-OA's conditions grammar has *no revocation primitive at all* — only
  `kind=` and `created_at<>` bounds, with "attestations are never revoked; a
  new attestation can revoke a prior one by including newer conditions."
  Under the current person-attested model, succession would require reissuing
  every attestation in the subgraph. Under (a) the attestation never changes
  on succession, because the subgraph is attested to the org. NIP-OA's
  weakest property stops mattering.
- (e) **approve, on grounds the root did not offer and with their offered
  ground rejected.** Reuse across three uses is evidence of economy, not of
  correctness. The test is shared *revocation semantics*, and the three uses
  pass it: revoker = grantor in every case (the identity of the grantor
  varies because the resource varies — a field, not a semantics difference);
  blast radius forward-only in every case (revoking history-read stops future
  extraction while already-labelled corpus rows stand per §4; revoking
  management authority stops future commissioning while a running node's seed
  is immutable mid-run); expiry identical. One kind with a scope tag.
- (f) **verified, and the root's premise is half wrong.** NIP-OA's owner field
  is a *pubkey* — the spec nowhere requires it be a person. An org or seat key
  substitutes with no spec change, so the org-attestation half needs **no new
  kind**. What genuinely needs its own kind is the *grant*, and that is the §4
  history-access-grant primitive we already owe ourselves — core-layer, ours,
  not a NIP-OA replacement. Allocate **42080 authority grant** / **42081 grant
  revocation**, mirroring the 42040/42041 request/verdict precedent (grants
  must be append-only so revocation is visible; a replaceable grant would
  clobber the evidence that authority once existed).
- **Honest limit to record:** nothing enforces a management grant today.
  Fractal does not read Nostr. Until commissioning emits an event, the grant
  is a governance fact like a decision-log row — which is fine, but must be
  *said*. Commissioning (C) is what gives it a checkable consumer.

### C. Commissioning (`35718844` item 2) — approve the composite, minus the spawn

The composition-vs-wrapping framing is accepted; the read-plane ruling does
not settle it. But the composite still *contains* the spawn, and §5.1 lists
the node-spawn API as out of scope. Split at the seam:

- **In scope (the commission):** resolve template version from the registry,
  generate the contract, commit for a pin, price caps, author the seed citing
  the pin, route for countersign, apply conditions, write the work order.
  None of it is a Fractal capability; all of it is untraceable today. It
  publishes a **kind 42070 commission** event — a signed, complete,
  executable specification.
- **Out of scope (the spawn):** `fractal node init` stays the operator's, via
  Fractal's CLI, exactly as §5.1's table already says.
- **The linkage the root wants, obtained without touching the spawn:** the
  instance's 42010 lifecycle event carries a `commission` reference. The
  instance→template-version link becomes a fact created by construction
  rather than a hand-typed line — bridge already emits 42010.
- **Authority question answered:** the registry is the record of what *runs*;
  the commission is the record of what was *authorized*. Different
  propositions, so no divergence — and where they differ, the difference is a
  **finding with a reader** (an instance running outside its commission),
  not a bug with none. That is precisely what approvals lacked.
- **Reframe the read plane honestly:** the read/write split was always about
  the *Fractal control plane*, not about the log. Publishing to the log is
  what every component does; evergreen may publish, it may not drive Fractal.
  §5.1 to say so, since the current wording invites the misreading.
- Approvals deferral **unchanged**, and now better-grounded.

### D. Evergreen absorbs intake (`2A25343E` item 2) — approve the merge

The root asked for the strongest case for separation. It is: a structural
boundary beats a behavioral discipline. It fails on two counts.

1. **The boundary is illusory.** An intake node that verifies requirements
   against template preconditions and produces a *validated* commission needs
   capability context and budget/priority context — i.e. owner context. You
   get two partially-informed nodes plus a synchronization problem, not
   isolation.
2. **The structural gate already exists at the right layer** — channel
   membership (§1, NIP-29). Node separation would be a second mechanism for
   what the channel does.

**Condition (converts the discipline into structure):** evergreen's outbound
surface to non-owners is built from log-derived fields only — the identical
rule §5.1 already binds the generated context surface with. "Write narrowly"
stops being a behavior and becomes the schema-enforced rule already on the
books.

### E. @mention front door (`2A25343E` item 3) — veto lifted, with conditions

- (b) is **the crux and it holds**: origin/effect asymmetry. The Buzz mention
  is the origin, the radio message the effect, the link explicit — one
  authoritative record. Confirmed by the rule in (C): Fractal asserts nothing
  about a mention that originated outside it.
- (a) new-capability-for-a-class-of-user accepted; (c) listener-not-harness
  and (d) routed-to-evergreen accepted as binding constraints, not colour.
- (f) conditions are **specified, not deferred**. Authorization: channel
  membership is the gate (§1), plus a per-requester allowance. Budget: (e)
  propose-not-dispose already closes the hole the root flagged — no spend
  moves before the human approves the commission, so a mention costs a read,
  not a run. Idempotency: key on the source event id; a commission is an
  upsert against that key, so redelivery is a no-op and an edited mention
  (new event id, per content-addressing) supersedes rather than duplicates.
- Registration ≠ mentionability, which resolves the UX lie without dropping
  fold-in (i): persistent roles may register profiles (identity, track
  record); only evergreen advertises an intake capability and answers.

### F. Fold-ins

- (i) persistent node ↔ Buzz agent at **role** level; ephemeral workers map to
  nothing. Approve — this is (A)(a) role-scoped identity seen from the other
  end, and Buzz's flat agent model accidentally enforcing §6.1 is worth
  recording.
- (ii) **vocabulary, and it falls out of the rulings rather than being
  invented**: **Fractal** = the platform; **tree/subgraph** = a running
  instance; **capability** = the durable asset (template + role identity +
  track record) — which is exactly what (A)(a) role-scoped identity plus
  (A)(d) version-scoped trust *define*. "Fractal" is retired from that third
  use.

## Post-Mortem

### Accomplished

All six rulings (A–F) landed as planned, plus the derived rule. `docs/DESIGN.md`
561 → 843 lines with eight decision-log rows, each naming its requesting
message and verdict. Verdicts `AAB14416` and `E4DC729B` — one per directive, so
each question maps to its answer rather than forcing the root to unpack a
merged reply. Shared wiki gained `platform_boundaries` (the working form of the
competing-assertion rule) and the three kind allocations; memory gained
`identity_and_commissioning`. Saved radio queue drained to empty.

The plan's central bet paid off: deriving one rule first and then applying it
six times, rather than arguing six cases. It also did work the plan did not
anticipate — it forced two corrections to my *own* prior reasoning (that "no
write path" had been readable as "never signs anything", and that "it wraps a
CLI" was never actually the decisive objection in the approvals ruling).

### Deviations

**§5.3 marked v2 explicitly, which the plan did not call for.** The plan said
"write conditions, not mechanisms" but said nothing about versioning, and
without a marker the whole subsection reads as v1 scope.

**One self-inflicted defect, caught at REVIEW.** I edited the §5.1 capability
table's **v1-ships** column to include "the commission itself", then eleven
lines later wrote that the commission path is v2. `evergreen` reads that table
as its scope, so it would have pulled v2 work into a v1 contract. Fixed by
restoring the v1 cell and adding an explicit note that the v1/v2 line runs
*through* that capability rather than around it. Worth naming the pattern: the
error came from editing a summary table to reflect a decision made in prose
elsewhere — the table and the prose were both "right" in isolation.

**Terminology collision I introduced.** Making "owner key" mean the *org* key
in §1 collided with §2's "the human owner" for template authorship. Rewritten
as three named keys with a stated convention.
