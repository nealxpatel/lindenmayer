## What the root asked, and where the real risk is

Root raised three questions and volunteered two deviations for overrule. My
reading is that **root's stated worry is not the actual hazard**: it asked
whether the dogfooding fixture is a §6.1 *privacy* leak, and the sharper
problem with that same requirement is *buildability*.

### Q(a) — scope `src,tests`, not `src` + `docs/`

**Uphold the root's deviation; overrule my own skeleton.** Two grounds:

1. Scopes are directory-granular, so granting `docs` grants write access to
   `DESIGN.md` — the manifest that governs the node. A first-ply node under
   architect review would hold commit rights over its own review bar. That
   is a governance inversion, and §6.4 (review is a veto, disagreement
   resolves as recorded conversation) presumes the reviewed cannot edit the
   standard.
2. The `docs/kinds/` precedent (row 419) already had to carve out a
   shared-directory arrangement for `core`, and it needed an explicit
   architect ruling to be safe. Repeating that pattern by default is how
   shared-directory friction becomes normal.

Residual gap to close rather than wave past: evergreen ships a user-facing
CLI and cannot document it under `src,tests`. Resolution — `--help` text and
module docstrings live in `src` and are the node's to write; any user-facing
doc *page* is mine, written from the node's escalation. Say so in the verdict
so "radio me if it needs a doc" is a defined path and not a bottleneck.

### Q(b) — dogfooding fixture: §6.1 concern? **Two-part answer.**

**On privacy: no concern, and the reason is structural.** Verified against
the wire formats rather than intent:

- 42020 is *run-grained by schema* — "no per-step fields exist anywhere in
  this event schema", privacy enforced at the wire level.
- 42010 carries `branch`/`status`/`run` + optional free-text `reason`.
- 42030 *is* the upward aggregate by construction.

So a surface derived from these kinds cannot carry more than the events do.
This holds **because** contract deliverable 2 is defined as derived from
deliverable 1 (the signed-log query surface) — that derivation is
load-bearing, not incidental. Add it as a condition: if the generator ever
reads Fractal's SQLite directly it acquires step-level detail that never
passed the wire-format filter, and the committed fixture becomes the leak.
Reinforcing: §7 already opts this dev tree into full transcript publication,
so the fixture is strictly *less* disclosure than what `transcripts/` publishes.

**On buildability: this is the finding.** Row 424 moved the core log to a
Lindenmayer relay whose selection and deployment §8 still lists as **open**;
`core/config.py:67` requires `relay_url` with no default; no relay endpoint
exists anywhere in the repo. If no relay is deployed, "run the generator over
this repo's live relay data" is a completion requirement the node **cannot
satisfy while alive** — the exact stranded-`exited` hazard the delegation rule
names, and one I already conditioned on in core row 419 (condition 3).

**Unresolved contradiction to settle in EXECUTE before ruling:** row 426
records dev-node v2 "registered as a signed 42050 (version event `0f8d0910`)"
and calls it "the registry's first real version bump" — which implies a live
relay. Either a relay is deployed and §8 is stale, or that registration was
against a mock and the row overstates it. **Verify; do not assume either way.**
This is precisely the negative-existence-claim trap from the compaction
episode (private note `1C1EF11D`) — except here the claim is *positive*
("live relay data"), and the same rule applies: it is a premise, not a finding.

Verdict branches on the evidence:
- Relay deployed → requirement **stands** (privacy answer above).
- Not deployed → **re-gate on the observable deliverable**: run the generator
  against the mock-relay fixture set and commit *that*, with the live run as a
  non-blocking follow-up once §8 closes. Preserves root's §7 dogfooding intent
  without staking the node's completion on an undeployed dependency.

### Q(c) — CLI-arg-path-under-test: per-contract or dev-node v3?

**v3 — a template change, and keep it in evergreen's contract meanwhile.**
The failure recurred across two *independent* nodes (bridge, registry), both
shipping CLIs whose argument path no test executed, both broken on first live
use. Two of two is a template gap, not an evergreen-specific risk; leaving it
per-contract means node four re-learns it by breaking. Registry's own work
makes the fix mechanical — a template bump is a new 42050 version event, and
v2 already did one (`0f8d0910`). Belt-and-braces: the clause stays in
evergreen's contract now, since v3 does not exist yet.

## Finding root did not raise: missing kind 38110

Deliverable 1 enumerates 42010, 42020, 42030, 42040/42041, 42050, 38150,
42060 — and **omits 38110**, the Node State Pointer. Its own kind doc says it
exists so "a dashboard resolves *current state of node X* in a single
addressable-kind query instead of replaying the full history from kind 42010
events." Deliverable 2 must surface "live subgraph state"; without 38110 the
generator replays whole 42010 chains to recover a status that one addressable
query returns. Condition: add 38110 to deliverable 1.
