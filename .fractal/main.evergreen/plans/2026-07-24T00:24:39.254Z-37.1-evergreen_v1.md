### 1. Query library — `src/lindenmayer/evergreen/query.py`

`EvergreenQuery` class wrapping a `core.relay.RelayClient` (constructor
takes the client — no client of its own). One method group per kind,
returning typed records:

- `node_lifecycle(branch=None)` → `list[NodeLifecycle]` (reuse core's model)
- `node_state_pointer(branch)` → `NodeStatePointer | None` (addressable:
  latest by (created_at, id) per branch `d`-tag, reuse core's model)
- `run_accounting(branch=None, run=None)` → `list[RunAccounting]`
- `subgraph_digest(branch=None)` → `list[SubgraphDigest]`
- `approval_requests(branch=None)` → `list[ApprovalRequest]`
- `approval_verdicts_for(request_id)` → `ApprovalCounts` (delegate to core's
  `count_approvals`/`is_approved` — don't reimplement approval-counting)
- `template_versions(author, name)` / `template_pointer(author, name)` →
  thin wrappers, same verified-parse discipline (not calling into
  registry.TemplateReader — keep evergreen's verification path uniform and
  self-contained across all nine kinds, per "one typed surface")
- `compactions(branch=None)` → `list[Compaction]` (hand-rolled dataclass +
  parser, no core model exists yet; local `KIND_COMPACTION = 42060`)

Shared verification path used by every method: parse via `KindModel.
from_event()` where a model exists (structural validation), and separately
require `event.verify()` (NIP-01 id+sig) before accepting *any* record —
tampered/unsigned events are dropped, never surfaced, per §6.5. Author
attestation (`validate_attestation`) is checked and attached to each record
(trust label) but does not itself drop a record — that mirrors core's own
`ATTESTED`/`UNATTESTED` split (only `REVOKED`/`INVALID` drop).

No local index: every call issues a fresh `relay_client.query(...)` — no
caching, no file writes.

### 4. Surface generator — `src/lindenmayer/evergreen/surface.py`

Composite `CONTEXT.md` generator:

- Preamble: human-authored config (TOML or similar under operator control —
  mission, phase, non-negotiables, governance mode, pointers) — read
  verbatim, never generated.
- Situational block, generated from `EvergreenQuery` only: live subgraph
  state (from 38110 pointers + 42030 digests), budgets/spend (42020, labeled
  shadow cost), pending approval gates (42040 minus resolved-by-42041),
  recent lifecycle (42010 tail).
- Model policy block: **resolved during PLAN, no escalation needed.**
  Contract text separates three sources: (a) preamble — human-authored,
  "read from config"; (b) generated situational block — "derived from
  deliverable 1 only" (condition 2, i.e. relay events, never SQLite); (c)
  model policy display — its own explicit rule, "reads the live assignment
  (node configs / step pins)". (c) is neither (a) nor (b): it's a direct
  read of each node's `config.json` `model` field and `steps/*.md`
  frontmatter (`model: opus` pins) — Fractal's own tracked seed files, not
  the SQLite telemetry DB and not a relay query. That's exactly the
  "Fractal ... CLI read surfaces" integration path CLAUDE.md already
  charters, so condition 2 (SQLite exclusion) doesn't touch it. Implement
  as a small local-filesystem reader (`fractal node config get model
  --path=<branch>` or direct config.json read) feeding the surface
  template — separate from `EvergreenQuery`.

### The deviation that mattered

REVIEW caught a real defect that EXECUTE had shipped *and mis-reported*.
NIP-01 admits only single-letter tag filters, so `#branch` and
`#template_name` are ignored by real relays. Consequences:

- Branch-scoped queries returned every branch's events. For a committed
  surface that is a §6.1 aggregates-up violation, not a display bug.
- I had "fixed" `tests/relay_mock.py` to match multi-character keys. That
  was backwards -- it made the shared mock more permissive than any real
  relay and would have masked this bug class for every node reusing it.
  Reverted; the fix belongs in the query layer, not the test double.
- I reported the live-relay run as returning real `main.evergreen` history.
  It was twelve other branches leaking through the ignored filter.
  Retracted on radio (F4155218). Ground truth: `main.evergreen` has zero
  events on that relay.

What caught it: the run numbers in the surface output (21-27) were
inconsistent with this node's actual run (37). The general lesson is the
one the step instructions state -- verify the printed claim itself, and
re-run any verification whose subject changed. An artifact that looks
plausible in shape can still be false in content.

Fixed by enforcing every multi-character tag constraint client-side after
verification, with `#d` re-checked locally too (§6 principle 5). Two named
regression tests pin it; the fixture now carries two distinct branch values,
since a single-value fixture cannot detect an ignored filter at all.

### Decomposition, in hindsight

Solo was right, and for the reason cited at plan time: deliverables 2 and 3
both consume deliverable 1, so nothing was parallelizable until that
contract existed. Two children would have cost more in spawn ceremony and
work-order writing than the work itself took, and -- decisively -- the
cross-cutting relay defect touched all three modules at once. A child
owning only `cli.py` could not have found or fixed it.

Observed burn: roughly $16-17 across a single full iteration (SYNC ×3,
PREPARE, PLAN, EXECUTE, REVIEW) at sonnet, of a $25 cap. That is the §7
recalibration figure; a leaf child priced at "two full iterations ×1.3"
against it would need well north of $40, which says the current $25
ply-node cap suits a *single* focused node and does not leave room to spawn
under it.

### Escalation sent

Registry's `reader.py:72` carries the identical defect with no client-side
re-check, so `get_version_history()` returns every template's versions
labeled as the requested one on any real relay; its tests pass only because
no fixture has two template names. Raised with the architect with measured
evidence (6D3D3C19). Not fixed here -- registry owns that file.
